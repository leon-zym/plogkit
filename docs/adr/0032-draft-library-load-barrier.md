# ADR 0032：草稿库以可重试加载屏障线性化权威快照

- 状态：已接受（2026-07-22）
- 修订：ADR 0030
- 关联：ADR 0022、0025、0031、[Issue #9](https://github.com/leon-zym/plogkit/issues/9)

## 背景

冷启动枚举与草稿 mutation 若独立推进，迟到的初始结果可能覆盖刚创建、保存或删除后的进程内列表。把单草稿读取异常一律归为损坏又会把暂时性 I/O 故障误导成可删除的数据损坏。首页不能通过合并旧数组、保留旧快照或把失败当作空列表补救这些不确定性。

## 决策

- Draft Library 的列表状态为 `uninitialized`、`loading`、`ready` 或 `storage-failed`。`load()` 对同一次加载尝试使用 single-flight：并发 caller 共享同一个 Promise；成功后安装一次完整快照，失败后不缓存失败 Promise，后续调用可以重新枚举。
- 加载中或失败时不暴露空、部分或旧快照作为可靠结果。Home 只触发加载、同步读取状态并订阅变化，不自行枚举、分类、排序或合并条目。
- UI 可以在加载期间开始选择照片等前置交互，但 create 的发布记录、save 的根记录替换和 delete 的删除标记都必须等待一次成功加载后才能提交。加载失败时不得产生这些 commit point；首次加载屏障不是所有磁盘操作的全局队列。
- mutation 的持久事务成功后，Draft Library 必须基于提交时的最新快照同步执行 functional update，再让操作 Promise 返回成功。不同草稿的完成顺序不得造成更新丢失，也不得出现“操作已成功但 caller 仍读到旧列表”的窗口。
- 同一 `DraftId` 的恢复、读取、保存、资产 transaction、缩略图提交、维护与删除继续串行；不同 `DraftId` 的磁盘工作允许并发。冷启动只在枚举多个草稿时施加受限并发。
- 只有相关 I/O 已得到确定结果，且恢复后仍能证明结构、引用或必需原图非法，条目才归类为损坏草稿。单个已确认损坏条目不阻塞其他条目进入可靠快照。
- 任一必需目录枚举、发布记录、删除标记、根记录、catalog 或原图存在性检查出现无法判定的 I/O 结果时，整个列表进入页面级 `storage-failed`；不把目标归类为损坏草稿，也不新增“暂时不可用”item。重试重新读取当前磁盘事实。
- mutation 提交结果无法判定时，同样撤销快照的可靠状态并进入页面级 storage failure；恢复成功前不以旧快照继续接受会改变草稿成员资格的操作。

## 影响与代价

- 单草稿的暂时性 I/O 故障可能暂时阻塞整个草稿库，但健康数据不会被误报成可删除的损坏草稿。
- 首次 mutation 可能等待冷启动扫描；快照就绪后仍保留跨草稿并发，不以全局 mutex 换取正确性。
- 快照、排序与故障分类集中在 Draft Library，使 Home 的状态模型保持浅薄，也让并发加载、失败重试和交错 mutation 可以直接通过 module contract 测试。
