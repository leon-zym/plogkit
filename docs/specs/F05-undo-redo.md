# F05 撤销与重做

- 状态：已实现
- 关联：[ADR 0004](../adr/0004-state-management-undo.md)

## 概述

对文档提交点提供有界的撤销与重做能力，覆盖所有编辑操作。

## 范围

- 撤销/重做按钮，作用于文档提交点。
- 栈深 30–50，越界丢弃最旧记录。
- 手势/滑杆过程不逐帧入栈。

## 非目标

- 操作历史列表 UI、跨会话撤销（重启后撤销栈清空，文档本身由 F06 恢复）。

## 需求与场景

### 需求 1：基本撤销重做

#### Scenario: 撤销添加文本

- GIVEN 用户刚添加了一个文本块
- WHEN 用户点击撤销
- THEN 该文本块从画布消失
- AND 点击重做后文本块恢复

#### Scenario: 连续多步撤销

- GIVEN 用户依次执行了加字、调背景色、调间距三次提交
- WHEN 用户连续点击撤销三次
- THEN 文档逐步回退到三次操作之前的状态，顺序与操作相反

### 需求 2：提交粒度

#### Scenario: 滑杆调整只产生一个撤销步

- GIVEN 用户按住间距滑杆连续拖动（产生大量中间值）
- WHEN 用户松开滑杆后点击一次撤销
- THEN 间距恢复到本次拖动开始前的值（整个拖动是一个撤销步）

### 需求 3：边界行为

#### Scenario: 栈空时撤销不可用

- GIVEN 编辑会话刚开始，无任何提交
- THEN 撤销按钮处于禁用态

#### Scenario: 新提交清空重做分支

- GIVEN 用户撤销了两步
- WHEN 用户执行一次新的编辑提交
- THEN 重做按钮变为禁用态（被撤销的分支不可再重做）

#### Scenario: 超出栈深丢弃最旧记录

- GIVEN 用户已连续执行超过栈深上限次数的提交
- WHEN 用户连续撤销到不能再撤销
- THEN 到达的状态是保留窗口内最旧的提交点，应用不崩溃
