# ADR 0007：导出管线：渲染/编码两段式与尺寸上限

- 状态：部分修订（2026-07-02 接受；见 [ADR 0023](0023-export-preset-catalog-and-pipeline.md)）
- 关联：ADR 0001、0003、0008、0009、0010

## 背景

导出是本项目最大的技术风险点。以 iPhone 24MP 原图为例：9 张竖向拼接按原尺寸导出约需 880MB RGBA 内存，且超出 GPU 纹理上限（16384px）；而按社交平台常用宽度（如 1242px）导出仅约 74MB，完全可行。"保留原始尺寸"在拼接场景是不可兑现的承诺。同时，HDR（ADR 0009）等诉求未来可能要求编码环节脱离 Skia。

## 决策

- 导出管线拆分为两段并接口化：
  - 渲染段：文档模型 → 像素（Skia，从文档渲染，绝不截屏预览画布）。
  - 编码段：像素 → 文件（当前为 Skia `encodeToBytes`，接口预留未来替换为平台原生编码实现）。
- 导出使用 CPU offscreen surface，绕开 GPU 纹理尺寸限制。
- 硬性尺寸上限：输出总像素 ≤ 64MP 且长边 ≤ 16384px；超限时按比例自动降级，并在导出预设语义中如实表述（"在上限内尽量保留"而非"保留原尺寸"）。
- 预览路径与导出路径分离：预览用低成本渲染（预览副本、屏幕分辨率），导出用全质量渲染（沙盒原图、目标分辨率）。

## 影响与代价

- 两段式接口是 HDR/Live Photo 未来重评估（ADR 0009）的逃生通道，避免届时推翻渲染主体。
- 上限值为工程判断的初值，可随实测调整；调整属参数变更，不需要新 ADR。
