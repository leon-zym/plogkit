# ADR 0028：草稿删除先提交待删除标记再异步清理

- 状态：已接受（2026-07-22）
- 关联：ADR 0022、0025、0027、[Issue #9](https://github.com/leon-zym/plogkit/issues/9)、[Issue #10](https://github.com/leon-zym/plogkit/issues/10)

## 背景

草稿目录包含统一文档、资产 catalog、多张原图及派生文件。将整个目录移动到 trash 可以先让草稿从列表消失再清理，但 Expo 的目录移动不提供本项目可以依赖的原子发布保证；跨文件系统退化或中途失败还会扩大到整个草稿 aggregate。直接递归删除则可能在中断后留下仍被列表发现的残缺草稿。

用户确认删除后不提供回收站或撤销，但需要明确、可恢复的删除 commit point：失败前保持完整可见，成功后不得因物理清理失败或进程终止重新出现。

## 决策

- 草稿删除先为目标 `DraftId` 提交一个路径隔离、结构可校验的待删除标记。标记使用 ADR 0025 的可恢复替换协议；只有合法标记成为 current 后，删除 transaction 才提交成功。
- 待删除标记是草稿库内部的删除意图，不是集中式草稿索引。列表、读取、打开、保存和 asset transaction 必须先检查该 `DraftId` 是否已提交删除；已标记草稿不再作为正常或损坏草稿展示，也不能重新打开。
- 标记提交成功后，草稿库幂等、best-effort 递归删除对应草稿目录。物理清理失败不改变已提交的用户删除结果；应用启动及后续安全维护入口继续清理。
- 只有确认草稿目录不存在后，才能移除待删除标记。进程在目录删除与标记移除之间终止时，下次维护确认目录已不存在并收敛标记；不得先移除标记再删除目录。
- 标记提交失败时不得开始物理删除，原草稿保持完整可见。未知、损坏或不符合 canonical `DraftId` 编码的标记不得隐藏草稿，并由 best-effort maintenance 隔离清理。
- 删除当前编辑会话所绑定的草稿时，必须先成功 flush 并建立阻止新编辑和 autosave 的 session barrier；删除提交成功后永久失效旧 handle。删除提交失败则恢复原会话，caller 不编排标记或文件清理顺序。
- 当前不移动或复制整个草稿目录到 trash，不提供用户可见回收站、短时撤销或最近删除列表。

## 影响与代价

- 用户可见删除只提交一个小型事实文件，不依赖大型目录移动的原子性；清理中断最多保留不可见数据，不会重新暴露半删除草稿。
- 待删除标记必须参与草稿库所有按 `DraftId` 的入口校验，并与同一草稿的其他 operation 串行；该复杂度集中在草稿库 module 内，不进入首页或当前编辑会话 caller。
- 物理空间可能在删除确认后暂时未释放。恢复性维护最终清理，清理故障不得报告为删除失败。
- 不提供用户恢复路径使 commit point 简单明确；若未来增加回收站，需要新的保留期限、恢复 transaction 与存储压力决策，不能把当前内部待删除状态直接暴露为产品能力。
