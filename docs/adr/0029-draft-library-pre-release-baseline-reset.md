# ADR 0029：草稿库产品化再次建立发布前持久化基线

- 状态：已接受（2026-07-22）
- 修订：ADR 0022
- 关联：ADR 0003、0027、0028、[Issue #9](https://github.com/leon-zym/plogkit/issues/9)

## 背景

ADR 0022 为草稿 aggregate 与当前编辑会话建立了第一个按 `DraftId` 工作的发布前持久化基线，并要求此后的 schema 变化恢复迁移纪律。草稿库产品化现在引入 ADR 0027 的草稿根记录、可枚举草稿元数据、草稿缩略图与 ADR 0028 的删除意图；产品仍未发布，没有需要保留的用户草稿数据。

为当前裸 `document.json` 布局实现 reader、转换器或双写，会留下只服务于内部开发数据的长期兼容分支，并迫使列表、损坏识别和删除 transaction 同时理解两套 aggregate 布局。

## 决策

- 草稿库产品化直接建立新的发布前持久化基线，只识别 ADR 0027 的草稿根记录及本轮确定的配套布局；不读取、迁移或双写当前开发版本的裸 `document.json` 草稿，也不恢复更早的 `projects/current` 单例数据。
- 不在生产代码中增加一次性 legacy reader、自动转换器、旧布局清理分支或 feature flag。开发机、模拟器、测试设备和 CI 在采用新基线后清空 App 沙盒重新开始。
- 本次 reset 只授权丢弃所有尚未发布的草稿持久化数据，不改变 `PlogDocument` 作为可迁移统一文档的职责，也不放宽正式发布后的 schema 纪律。
- 新基线确定后，草稿根记录、统一文档、资产 catalog 或其他已发布持久事实的未来不兼容变更，必须提升各自 schema version 并提供迁移；再次丢弃数据需要新的显式 ADR，不能援引本决策。

## 影响与代价

- 草稿库只有一套持久化布局，列表、恢复、损坏展示和删除 transaction 不承担内部开发历史的兼容复杂度。
- 已安装开发构建中的现有草稿会被明确丢弃，升级验证必须从干净沙盒开始；这不影响尚不存在的正式用户数据。
- ADR 0022 关于本轮之前 baseline reset 的历史仍保留；其“此后迁移”规则由本 ADR 对草稿库产品化这一轮作一次明确例外，新的基线完成后继续生效。
