# ADR 0033：草稿删除使用独立外部标记并由当前编辑会话协调

- 状态：已接受（2026-07-22）
- 修订：ADR 0022、0028
- 关联：ADR 0025、0027、0029、0032、[Issue #9](https://github.com/leon-zym/plogkit/issues/9)

## 背景

ADR 0028 以待删除标记建立删除 commit point，但让一个没有旧值的写一次标记复用 `current`、`backup`、`temp` 替换协议，会增加无意义的恢复状态。把全部待删除 `DraftId` 写入一个全局文件，则要求每次删除和清理重写共同事实、全局串行并承受单文件损坏扩大到所有删除结果。删除当前草稿还需要协调编辑、autosave、资产操作与 handle 生命周期，UI caller 无法安全编排这些顺序。

## 决策

- 每个草稿使用一个位于草稿目录之外的扁平标记：`deletions/<canonical-draft-id>.json`。标记具有独立 schema version，保存 `DraftId`，且内容必须与路径编码一致；有效标记写一次且不可变。
- 标记直接写入 canonical 路径，不使用子目录、全局待删除列表或 `current`、`backup`、`temp`。提交前先读取 canonical 路径：已有有效标记按幂等成功返回且永不改写；可读取但非法的未提交残留只有在清理后才能重新尝试。无论写入调用成功或报错，草稿库都重新读取并严格校验：有效标记表示逻辑删除已提交；明确不存在或可读取但非法表示未提交；无法读取表示结果未知。
- 只有有效标记是删除 commit point。提交后，列表、打开、保存、资产 transaction 和缩略图提交都不再把目标作为草稿处理；随后幂等、best-effort 递归清理草稿目录。只有确认草稿目录不存在后才能移除标记。
- 非法标记不得隐藏草稿，可由维护流程清理。结果未知时不重复盲目删除，也不恢复可能已经失效的会话；草稿库进入页面级 storage failure，重试只重新判定磁盘事实。
- `CurrentEditingSession.delete(draftId)` 是应用层唯一删除入口，Draft Library 的删除 transaction 保持内部能力。删除非当前草稿或损坏草稿时，会话 module 直接委托；删除当前草稿时，open、flush、资产变更与 delete 共用 lifecycle gate。
- 当前草稿删除先阻止新编辑、autosave 与资产操作，等待在途操作并成功 flush，再提交删除标记。标记有效时永久失效旧 handle；明确未提交且草稿仍完整时解除 barrier 并恢复原会话；结果未知时保持 barrier，直到重试或冷启动恢复出确定事实。
- 同一 `DraftId` 的删除与所有其他持久操作继续串行，避免删除后陈旧保存重新写入。不同草稿的删除可以独立提交。
- 未来多选删除或全部清除可以固定确认时的 `DraftId` 集合，逐草稿提交并允许部分成功。只有未来要求批次 all-or-nothing 或进程重启后自动完成整批时，才新增独立 batch manifest；当前不提前引入。

## 影响与代价

- 每草稿标记让并发、恢复和清理保持故障隔离；全部清除需要多个小写入，但不会形成新的全局权威索引。
- 当前会话吸收删除 lifecycle 后，Home 不再掌握 `flush → delete → invalidate` 顺序，陈旧 handle 和 autosave 无法在删除窗口复活草稿。
- 结果未知会暂时冻结当前会话并阻塞草稿库，代价是不会在删除已经提交时错误恢复编辑，也不会在删除未提交时谎报成功。
