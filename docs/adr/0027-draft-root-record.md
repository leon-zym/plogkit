# ADR 0027：草稿身份、元数据与统一文档共用可恢复根记录

- 状态：已接受（2026-07-22）
- 关联：ADR 0003、0022、0025、[Issue #9](https://github.com/leon-zym/plogkit/issues/9)

## 背景

草稿库需要持久化 `DraftId`、创建时间和最近编辑时间，并保证最近编辑时间与成功保存的统一文档一致。继续把统一文档与草稿元数据保存为两个独立事实文件，需要额外的共同 revision、commit marker 或跨文件恢复状态机；把这些字段直接加入 `PlogDocument`，则会让草稿身份和列表生命周期污染编辑、渲染与导出的内容模型。

资产 catalog、草稿缩略图和统一文档具有不同生命周期。catalog 可以暂时包含未被最新文档引用、但仍供当前编辑会话撤销重做使用的导入资产；草稿缩略图是允许滞后、缺失和重建的派生数据。将完整草稿 aggregate 塞入同一个物理文件，会把无需共同提交的状态耦合到同一故障域。

## 决策

- 每个草稿使用一个带独立 schema version 的可恢复根记录，物理上共同保存 `DraftId`、草稿元数据与嵌套的 `PlogDocument`。草稿元数据至少包含不可变的创建时间和只随新统一文档版本成功保存而变化的最近编辑时间。
- `DraftId` 和草稿元数据不进入 `PlogDocument`；草稿根记录 schema、统一文档 schema 与资产 catalog schema 分别演进和迁移。
- 草稿创建发布根记录时同时产生创建时间与初始最近编辑时间。草稿保存以整个根记录为一个恢复单元；保存 caller 只提交统一文档，不提供时间或编排元数据写入。打开、导出、缩略图重建、资产 ingest 与非活跃维护不改变最近编辑时间，语义 no-op 保存也不改变。
- 资产 catalog 继续作为独立的可恢复事实文件，并沿用“先发布资产与 catalog，后让统一文档引用；先解除文档引用，后在非活跃维护中压缩 catalog”的 transaction 顺序。原图、拍摄信息 sidecar、编辑预览和草稿缩略图仍按各自生命周期独立保存。
- 草稿缩略图在根记录保存成功后异步、best-effort 重建；失败不改变保存结果，旧缩略图可以继续用于草稿库展示。
- 草稿目录名继续使用 `DraftId` 的可逆、路径安全、canonical 编码；根记录内的 `DraftId` 必须与目录编码一致。根记录完全损坏时，草稿库仍可从合法目录编码恢复身份，以损坏草稿形式枚举和删除；无法解码的未知目录不作为用户草稿展示。
- 草稿根记录沿用 ADR 0025 的 `current`、`backup`、`temp` 可恢复替换协议。同一 `DraftId` 的枚举恢复、读取、保存、asset transaction、缩略图提交、维护与删除继续在草稿库内部串行。
- 当前不引入独立 metadata repository、集中式草稿索引、跨文件两阶段提交或持久草稿版本日志。若未来出现修改同一草稿的多进程 writer、用户可见的持久版本历史，或统一文档不再适合整体 JSON 替换，再在草稿库 interface 后重新评估不可变 revision 与 CAS；这些能力不提前进入当前持久化协议。

## 影响与代价

- 统一文档与最近编辑时间通过一次可恢复替换获得强一致性，当前编辑会话和首页无需理解持久化布局、时间生成或恢复顺序。
- 草稿身份与列表事实获得明确持久化归属，同时统一文档继续保持纯粹的作品内容模型；未来增加草稿级字段不必提升 document schema。
- 列表需要读取每个草稿的根记录，而不是只读取更小的独立 metadata 文件。当前统一文档只保存小型 JSON，草稿库通过受限并发、虚拟化 Grid 和按需缩略图承担该成本。
- 根记录损坏时可能无法取得排序时间；草稿库按产品规则将缺少最近编辑时间的损坏项置顶，并使用旧缩略图或损坏占位图展示。
- 本决策依赖单进程 writer 与小型统一文档。它以更低的当前状态空间和维护成本，换取不提前提供持久版本历史与多进程写入能力。
