# ADR 0030：草稿库以冷启动枚举构建进程内权威快照

- 状态：已接受（2026-07-22）
- 修订：ADR 0022、0025
- 关联：ADR 0027、0028、0029、[Issue #9](https://github.com/leon-zym/plogkit/issues/9)

## 背景

草稿库需要按最近编辑时间全局排序，不设人为数量上限，并把自动恢复后仍无法安全打开的损坏草稿作为明确异常项展示。集中式权威索引可以减少冷启动读取，却需要让每次创建、保存、删除和恢复同时维护跨草稿事实；索引与草稿 aggregate 之间的半提交或索引损坏会扩大故障范围。

另一方面，每次返回首页都重新枚举磁盘没有意义：PlogKit 当前只有单进程 writer，草稿只能通过 Draft Library module 改变。把完整校验推迟到 Grid 可见区域可以降低首屏 I/O，但会引入未校验 item、滚动预取和状态跳变，并可能让损坏草稿短暂表现为正常草稿。

## 决策

- 不建立集中式权威草稿索引。应用冷启动时枚举 canonical 草稿目录与待删除标记，以受限并发对每个草稿执行一次确定性的恢复和完整性校验，再构建稳定列表快照。
- 恢复先收敛草稿根记录和资产 catalog 的 `current`、`backup`、`temp` 状态，再校验根记录、`DraftId`、统一文档、catalog、文档引用及所需原图存在性。该过程只读取小型持久事实和文件存在性，不读取图片 bytes，不解码或重建缩略图。
- 自动恢复后通过完整性校验的条目作为正常草稿；仍无法安全打开的条目作为损坏草稿返回。损坏条目不是可打开的草稿，但保留在 Grid 中供用户识别和删除；有旧草稿缩略图时继续使用，否则展示损坏占位图，两者均叠加明确警告状态。
- 缺少最近编辑时间的条目排在顶部，其余按最近编辑时间倒序；相同排序事实使用稳定的内部 tie-breaker，避免列表抖动。排序和损坏分类由 Draft Library implementation 拥有，首页不解释持久化失败或拼装顺序。
- 冷启动只有在草稿根目录整体无法枚举或无法产生可靠快照时返回页面级 storage failure；单个草稿失败不得让整个列表失败。页面级故障可以由用户重试，已判定损坏的单项不提供同进程手动重试。
- 首次快照完成后，Draft Library 在同一进程内持有权威列表快照。create、save、delete 及损坏分类变化在对应 transaction 成功后原子更新快照；返回首页直接读取该快照，不重新扫描磁盘。下次进程启动重新枚举并恢复。
- 首页使用虚拟化 Grid，草稿缩略图按可见范围加载；完整性校验不延迟到可见 item。当前不增加未校验 item 状态、滚动预取校验、派生全局列表缓存或后台全库刷新。
- 取消持久化最近草稿定位器和“继续上次编辑”快捷入口。应用启动不自动打开草稿；当前进程内的当前编辑会话仍由 Current Editing Session module 持有，用户通过按最近编辑时间排序的 Grid 重新打开草稿。
- 若真实设备验证表明全量根记录与完整性校验成为冷启动瓶颈，可以在保持目录和草稿根记录为事实源的前提下新增可重建、非权威列表缓存；不得在没有测量证据时把缓存升级为新的写入 transaction 或 external seam。

## 影响与代价

- 冷启动工作量随草稿数和引用资产数线性增长，但每个进程只执行一次，并避免读取图片内容与不可见缩略图。设备存储先于任意草稿数量上限约束实际规模。
- 首页获得一次完成、顺序稳定且已分类的快照，不需要处理 item 从未校验到损坏的状态跳变；同进程返回首页没有重复 I/O。
- 损坏草稿成为正常列表模型中的显式异常 variant，修订了 ADR 0025“列表只展示完整 aggregate”的限制，但仍不会以残缺内容进入 Editor。
- 移除最近草稿定位器减少了一个可能指向已删除或损坏草稿的持久事实；作为代价，打开旧草稿但未形成新文档版本不会改变排序，用户仍从原 Grid 位置重新进入。
- 不使用权威索引保持了每草稿故障隔离和 transaction locality；如果未来规模测量要求缓存，缓存必须允许丢弃后从草稿目录完整重建。
