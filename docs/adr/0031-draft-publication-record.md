# ADR 0031：草稿创建以不可变发布记录作为提交点

- 状态：已接受（2026-07-22）
- 修订：ADR 0022、0025、0027、0030
- 关联：ADR 0029、[Issue #9](https://github.com/leon-zym/plogkit/issues/9)

## 背景

Expo 不为草稿目录移动提供本项目可以依赖的原子发布保证。目标目录当前完整可读，也不能持久区分“创建失败后未清理的完整残留”与“曾成功创建、后来根记录完全损坏的用户草稿”。只依赖 aggregate 完整性会迫使草稿库在隐藏失败残留与保留损坏草稿之间二选一；建立集中式发布索引则会把所有草稿耦合进共同写入事务与故障域。

## 决策

- 每个最终草稿目录包含一个写一次的 `publication.json`。记录具有独立 schema version，保存 `DraftId`，且内容必须与 canonical 目录编码一致；它是内部发布事实，不是新的草稿身份、用户数据或列表索引。
- staging 不包含发布记录。创建先把 aggregate 发布到最终目录，恢复并完整校验草稿根记录、catalog、文档引用与必需原图，最后才在最终目录直接写入 canonical 发布记录。有效发布记录是创建成功的唯一 commit point。
- 发布记录没有需要保留的旧值，不使用 `current`、`backup`、`temp` 可恢复替换。无论写入调用成功或报错，草稿库都重新读取并严格校验：记录有效表示创建已提交；明确不存在或可读取但非法表示未提交；读取结果无法判定时返回 storage failure，不猜测结果。
- 只有有效发布记录确认的目录属于用户草稿。其 aggregate 有效时枚举为正常草稿；aggregate 恢复后仍非法时枚举为损坏草稿。没有有效发布记录的最终目录是内部创建残留，不进入 Grid，由安全维护入口 best-effort 清理。
- 创建始终生成新的 `DraftId`，并在发布前确认目标目录不存在；身份碰撞或目标已存在时创建失败，不覆盖、合并或接管已有草稿。
- 有效发布记录随草稿目录保留，直到外部删除标记提交后随目录物理清理。它不随文档保存、catalog 变更、缩略图重建或维护而改写。

## 影响与代价

- 每个草稿增加一个很小的持久事实，换取创建结果在进程终止后仍可判定，并让根记录完全损坏的既有草稿继续作为损坏草稿出现。
- aggregate 移动、完整校验与发布记录提交形成明确顺序；失败清理不再承担区分用户草稿与内部残留的职责。
- 发布记录无法可靠读取时，草稿库不能安装可靠快照。该保守失败会暂时阻塞列表，但不会把失败创建暴露给用户或静默丢弃已提交草稿。
